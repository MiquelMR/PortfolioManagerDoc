Code Style

Naming and organizational consistency rules

# Data Layer: Services and Repositories

General Rules

  • - Both services and repositories will follow the same ruleset.
  • - Methods should appear in the same order as listed in this document.
  • - The order of appearance in the interface and the implementing class should be the same.
  • - Methods should contain the full property name, without abbreviations.

# Method Naming Ruleset and Order

  • Operations on a single entity:

    VerbEntityByProperty → Example: GetUserByUserId

  • Operations on multiple entities:

    VerbEntities → Example: GetFinancialAssets

  • Operations on multiple entities with filter:

    VerbEntitiesByProperty → Example: GetPortfoliosByUserEmail

  • Utility methods:

    VerbEntitiesByProperty → Example: ExistsByUserEmail

Methods should appear in the following order:

  1. Get single entity
  2. Get multiple entities
  3. Get multiple entities by property
  4. Create entity
  5. Update entity
  6. Delete entity
  7. Public utilities
  8. Private utilities

Data Organization Ruleset

  • - One service per entity
  • - Use interfaces for every service
  • - ViewModels may use several services
  • - One ViewModel per page
  • - Several components per page
  • - Use scoped Dependency Injections

    • Example: HomeOverview and HomeCreatePortfolio bind data and send EventCallbacks from the Home code-behind, which has a HomeViewModel that uses UserService and PortfolioService.


  • - Page and component naming is based on the view's purpose. Example: Home.razor for the home page.
  • - Code-behind and ViewModel naming is derived from page and component names. Example: HomeViewModel for the home page.

ViewModel and Code-Behind Ruleset

Code in ViewModels, pages, and components' code-behind should appear in the following order:

  1. Dependencies (e.g., services, NavigationManager, ViewModel, etc.)
  2. Properties
  3. Delegates
  4. Private fields
  5. Initialize method
  6. Public methods
  7. Private methods
  • Every property or field declaration must be in a single line, including annotations.
  • Example: [Parameter] public List<FinancialAsset> FinancialAssets { get; set; } = [];
